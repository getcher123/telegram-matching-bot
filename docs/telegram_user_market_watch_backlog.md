# Поэтапный backlog разработки проекта `tg-market-watch`

**Документ:** дополнительный план разработки к первому описанию проекта `telegram_user_market_watch_project.md`  
**Назначение:** превратить архитектурное описание в набор задач, которые можно распределять между субагентами разработки.  
**Проект:** FastAPI-приложение, подключающееся к Telegram как user-аккаунт, читающее сообщения в разрешённых группах, применяющее детерминированные YAML-правила и отправляющее алерты выбранному Telegram-пользователю.  
**Начальные сценарии мониторинга:**

1. Продажа любого телевизора с диагональю 50+ дюймов.
2. Продажа MacBook с чипом M4 Pro.
3. Продажа AirPods Pro 2.

---

## 1. Как использовать этот backlog

Каждая задача оформлена так, чтобы её можно было отдать отдельному субагенту без дополнительной расшифровки. В карточках задач используются единые поля:

- **ID** — стабильный идентификатор задачи.
- **Тип** — feature, infra, qa, security, docs, research, refactor.
- **Приоритет** — P0, P1, P2.
- **Рекомендуемый субагент** — профиль исполнителя.
- **Цель** — что должно быть достигнуто.
- **Требования** — обязательные функциональные и технические условия.
- **Выходные артефакты** — конкретные файлы, модули, API, тесты или документы.
- **Критерии приёмки** — проверяемые условия, по которым задача считается выполненной.
- **Зависимости** — задачи, которые должны быть завершены ранее.

Backlog построен по этапам. Этап считается завершённым только после прохождения его **gate-критериев**.

## 3. Глобальные требования ко всему проекту

### 3.1. Обязательные принципы

1. **Детерминированность:** один и тот же текст при одном и том же YAML-конфиге всегда даёт одинаковый результат.
2. **Объяснимость:** каждое совпадение содержит правило, признаки, нормализованный текстовый фрагмент, числовые значения и причину решения.
3. **Безопасная работа с Telegram:** приложение обрабатывает только чаты, куда user-аккаунт был добавлен законным способом. Автоматическое вступление в закрытые группы, обход ограничений и массовая рассылка не реализуются.
4. **Атомарная перезагрузка конфига:** невалидный YAML не ломает уже работающие правила.
5. **Защита от дублей:** одно и то же сообщение не должно генерировать повторный алерт без явной причины.
6. **Тестируемость:** нормализация, извлечение сущностей и rule engine покрываются unit- и regression-тестами.
7. **Секреты не логируются:** Telegram session, номера телефонов, коды авторизации, 2FA-пароли и токены не попадают в логи.
8. **MVP без LLM в рантайме:** решение о совпадении принимает только детерминированная логика.

### 3.2. Базовый Definition of Done для каждой задачи

Задача считается выполненной, если:

- код соответствует контрактам проекта;
- добавлены или обновлены тесты;
- тесты, относящиеся к задаче, проходят локально;
- ошибки обрабатываются явно;
- нет логирования секретов;
- документация обновлена, если задача меняет пользовательское поведение, API, YAML-конфиг или схему БД;
- результат можно проверить по критериям приёмки без устных пояснений исполнителя.

### BL-0003. Описать доменные модели проекта

**Тип:** feature  
**Приоритет:** P0  
**Рекомендуемый субагент:** `ARCH`

**Цель:** создать единые модели данных, которыми будут пользоваться Telegram, нормализация, rule engine, storage и API.

**Требования:**

- Описать модели: `RawMessage`, `NormalizedMessage`, `ExtractedEntities`, `RuleDecision`, `EvidenceItem`, `MatchRecord`, `AlertRecord`, `ChatInfo`, `ConfigVersion`.
- Модели должны быть сериализуемы в JSON.
- Модели должны иметь стабильные поля для сохранения в БД и выдачи через API.
- Для `RuleDecision` должны быть предусмотрены статусы: `MATCH`, `NO_MATCH`, `REJECTED_BY_NEGATIVE`, `INSUFFICIENT_EVIDENCE`, `SKIPPED_BY_SCOPE`.
- Для `EvidenceItem` должны быть поля: `kind`, `name`, `value`, `source_text`, `normalized_text`, `start_token`, `end_token`, `confidence`.
- Поле `confidence` используется как объяснительная метка качества признака, а не как вероятностное ML-решение.

**Выходные артефакты:**

- `app/core/models.py`
- тесты сериализации моделей
- документация форматов в `docs/domain_models.md`

**Критерии приёмки:**

- Все модели проходят JSON round-trip: объект сериализуется и восстанавливается без потери ключевых полей.
- `RuleDecision` содержит список evidence и человекочитаемое объяснение.
- Модели не импортируют Telethon, FastAPI или SQLAlchemy напрямую.
- Модели можно использовать в unit-тестах без запуска приложения.

**Зависимости:** BL-0001.

## 6. Этап 1. FastAPI control plane

### BL-0101. Реализовать каркас FastAPI-приложения с lifespan

**Тип:** feature  
**Приоритет:** P0  
**Рекомендуемый субагент:** `API`

**Цель:** создать запускаемый FastAPI-сервис, который корректно стартует и завершает внутренние компоненты.

**Требования:**

- Использовать lifespan-контекст для инициализации `ConfigService`, `StorageService`, `TelegramClientService` и `AlertDispatcher`.
- На старте сервис должен валидировать базовые настройки.
- На остановке сервис должен корректно закрывать соединения и фоновые задачи.
- Состояние компонентов должно быть доступно через общий `AppState`.

**Выходные артефакты:**

- `app/main.py`
- `app/core/app_state.py`
- `app/core/settings.py`
- тест запуска приложения через тестовый клиент

**Критерии приёмки:**

- Приложение стартует без Telegram-подключения в тестовом режиме.
- Lifespan вызывает start и stop у зарегистрированных сервисов.
- Ошибка старта компонента переводит приложение в диагностируемое состояние.
- При остановке не остаётся незавершённых фоновых задач в тестах.

**Зависимости:** BL-0003.

### BL-0104. Ввести простую защиту административного API

**Тип:** security  
**Приоритет:** P1  
**Рекомендуемый субагент:** `API`, `DEVOPS`

**Цель:** защитить управляющие endpoints от случайного доступа.

**Требования:**

- Для endpoints, изменяющих состояние, добавить проверку административного ключа.
- Ключ задаётся через переменную окружения.
- `GET /health` остаётся открытым.
- `GET /status` может работать в безопасном сокращённом режиме без ключа и полном режиме с ключом.

**Выходные артефакты:**

- middleware или dependency для API-key проверки
- тесты успешного и неуспешного доступа
- описание в `docs/api.md`

**Критерии приёмки:**

- `POST` endpoints без ключа возвращают HTTP 401 или HTTP 403.
- Валидный ключ даёт доступ к защищённым endpoints.
- Ключ не выводится в логи.
- В тестах есть проверка, что `GET /health` остаётся доступным.

**Зависимости:** BL-0101.

### BL-0202. Реализовать flow авторизации Telegram user-аккаунта

- Реализовать `POST /auth/send-code`.
- Реализовать `POST /auth/confirm-code`.
- Реализовать `POST /auth/confirm-2fa`.
- Поддержать состояние ожидания кода и состояние ожидания 2FA.
- Не сохранять auth code и 2FA-пароль в БД.
- Не логировать auth code и 2FA-пароль.
- После успешной авторизации session-файл хранится в защищённом каталоге приложения.

- тесты flow с mock Telegram-клиентом

- При успешном вводе 2FA API возвращает состояние `authorized`.
- Ошибочный код возвращает контролируемую ошибку без падения приложения.

- Сообщения, matches, alerts и offsets сохраняются.
- Дубли не создают повторные alerts.
- История доступна через API.

### Gate 7. Alerts готовы

- Match создаёт Telegram alert.
- Alert содержит ссылку или честный fallback.
- Retry и rate-limit работают.
- Статусы доставки сохраняются.

### Gate 8. MVP готов

- E2E сценарии трёх правил проходят.
- Regression corpus зелёный.
- Docker запуск работает.
- Runbook готов.
- MVP checklist закрыт.
